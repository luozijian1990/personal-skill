# 贡献指南

本仓库定位为经过筛选、改造和组合的 Agent Skills 合集，欢迎以下三类贡献：

- 新增 Skill；
- 改进现有 Skill（修订、扩展、文档化）；
- 修复问题（坏链、过时命令、错别字、依赖错位等）。

提交前请阅读以下最低要求。Pull Request 必须满足这些要求才会被合入。

## 1. 单 Skill 自包含

- 每个 Skill 位于 `skills/<name>/` 单独目录，目录名使用清晰、易触发的英文短语；
- 资源（脚本、模板、参考、示例）放在该 Skill 目录下，**禁止跨 Skill 相对引用**；
- 公共脚本若必须共享，先在 issue 中讨论再决定放置位置；
- 不要在 Skill 内引用维护者的本地绝对路径或仅本机存在的命令。

## 2. 来源与许可证

| 类型 | 要求 |
|---|---|
| 原创 Skill | 在 PR 描述中明确声明原创，并接受根 `LICENSE`（MIT）覆盖 |
| 改编 / 收录的第三方 Skill | 提供上游仓库 URL、具体目录或文件路径、上游许可证 SPDX 标识、改动摘要；在 `THIRD_PARTY_NOTICES.md` 增加对应条目；如上游为 MIT 等需保留版权声明，需将上游 `LICENSE` 原文复制到 `THIRD_PARTY_LICENSES/<owner>-<repo>-LICENSE.txt`（或按 Anthropic 风格随 Skill 目录保留） |
| 上游无 LICENSE 文件的内容 | 在合入前必须取得上游作者明确的再分发授权（issue 回复、邮件、commit 注释、社交平台 DM 截图等），证据归档到 `docs/permissions/`；否则不接受合入 |
| 上游 LICENSE 不允许再分发 | 不接受合入 |

不接受以下任意情况：
- 仅以"仅供学习"作为再分发依据；
- 仅以 README 徽章替代实际 LICENSE 文件；
- 修改、剔除或替换上游版权声明、NOTICE 文件。

## 3. 验证说明

- 在 PR 描述中说明：
  - 测试环境（Agent 名称、模型版本、操作系统、关键依赖版本）；
  - 至少一组触发示例（可拷贝粘贴的提示词或调用方式）；
  - 实际验证结果（截图、Markdown 报告、命令输出片段均可，禁止仅写"应该可用"）；
- 若 Skill 涉及外部服务（云服务、第三方 API）或破坏性操作（写盘、改库、外发请求），**必须**说明清楚副作用与所需凭据，并避免在示例中泄露真实凭据；
- 若无法在公开环境复现验证，需在 PR 中明确说明限制，并补一份最小可演示的样例。

## 4. Pull Request 检查清单

提交前请在 PR 描述中逐项确认：

- [ ] `SKILL.md` 的 frontmatter 至少包含 `name`、`description` 字段；`description` 描述清晰，能够准确触发；
- [ ] 所有引用使用相对路径（`./xxx`、`../xxx`），未引用维护者本地绝对路径或仅本机存在的命令；
- [ ] 所有外部依赖（脚本、二进制、API、凭据）在 `SKILL.md` 中明确列出，并说明如何安装或获取；
- [ ] 示例、脚本、配置中不含真实凭据、Token、内部域名、企业内部信息；演示数据使用占位符；
- [ ] 新增 / 改编第三方内容时，已在 `THIRD_PARTY_NOTICES.md` 增加条目，并按需在 `THIRD_PARTY_LICENSES/` 增加上游 LICENSE 原文；
- [ ] 根 `README.md` 的对应分类已添加新 Skill 的链接和一句话简介；分类不存在时另起一段并说明理由；
- [ ] 已按上节说明在 PR 描述中给出测试环境、触发示例与实际验证结果。

## 5. 风格与约定

- Skill 文档默认中文，必要时可英文混写，但 `name` 必须为英文短语；
- 避免在 Skill 中复述 Agent 平台的通用入门内容；聚焦本 Skill 解决的问题、触发条件、关键参考；
- 单个 Skill 目录文件数控制在合理范围；引用文档放 `references/`，可执行脚本放 `scripts/`，模板放 `templates/`。

## 6. 反馈与讨论

- 想法 / 设计层面的讨论请走 issue；
- 涉及上游许可证、版权、再分发授权的疑问，请在 issue 中明确给出上游链接和具体疑问，便于追溯；
- 紧急修复（链接失效、依赖断裂）欢迎直接发 PR，并在标题前缀 `fix:` 标明。
