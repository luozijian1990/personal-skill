# learning-notes-builder

把一批同主题的源文件（.md / .txt / .html / .rst）合成为一份系统化的中文学习笔记 markdown。产出的笔记结构（`##` 章节 + `###` 小节）恰好是下游 [[learning-roadmap]] 的输入格式，两者闭环。

## 何时用 / 何时不用

**适用**：
- 用户已经收集好一批同主题素材（爬虫 dump、极客时间下载、官方文档 clone、剪藏的零散 md/txt 等），文件量大致 5-100 个
- 素材按目录或文件名编号组织（如 `01-入门/`, `01--xxx.txt`），或至少能按文件名自然排序
- 用户想要的是**学习型笔记**（讲解 + 图 + 代码），不是参考手册或速查表

**不适用**：
- 单个大文件（直接读就行，不用 skill）
- 完全无序的素材（用户得先自己整理目录）
- 散文 / 故事 / 新闻类内容（用 baoyu-markdown-to-html 之类的工具更合适）
- 用户要求"提取关键信息" / "总结要点"（用普通摘要就够）

## 五阶段工作流

整体节奏：**先扫描 → 先大纲 → 用户确认 → 再动笔**。每个 checkpoint 都要停下来等用户点头，绝不一口气从扫描跑到 5000 行正文——那是 prompt 不是 skill。

### Phase 1：扫描 + 提议元信息

```bash
python3 ~/.skills/learning-notes-builder/scripts/scan_sources.py <材料目录>
```

输出含：`topic_guess`、文件总数、章节数、推荐输出文件名 + 起点大纲（每个文件附 160 字预览）。

读完输出后，给用户**一段简短判断**（4-5 行）：

```
我看了下材料：
- 主题：Kafka（来自目录名 + 文件内容反复出现的术语）
- 规模：44 文件 / 7 章节
- 建议读者：运维 / SRE（基于文件偏运维内容；如不对告诉我）
- 建议输出：kafka/Kafka_Learning_Notes.md
- 建议大纲文件：kafka/Kafka_Learning_Notes_Outline.md
没问题的话我开始写大纲。
```

等用户点头或修正（最常见修正：读者定位、文件名）。**不要用 AskUserQuestion**——就让用户回一句话即可，灵活。

### Phase 2：生成大纲文件

基于扫描输出 + 必要时抽样阅读 2-3 个关键文件，写一份大纲到 `<output_stem>_Outline.md`。约束：

1. **章节 / 小节名要根据内容重写**，不要直接套用文件名（"01---消息引擎系统ABC" → "消息引擎系统是什么"）
2. **每个 `###` 小节后用 HTML 注释挂源文件映射**，方便后续核对。注释**必须独占一行**，紧跟 `### ` 下方——绝不能跟标题同行，否则下游 parser 会把注释当作标题的一部分显示出来：
   ```markdown
   ### 消息引擎系统是什么
   <!-- src: 01-Kafka入门/01---消息引擎系统ABC.txt -->
   ```
3. 一个小节可以合并多个源文件（如果它们讲同一个主题），用 `; ` 分隔多个 src
4. 不写正文，只写标题骨架 + 源文件映射

写完后告诉用户：
```
大纲已写到 kafka/Kafka_Learning_Notes_Outline.md（X 章 / Y 节）。
你可以直接编辑这个文件（合并 / 拆分 / 改名 / 调顺序），改完告诉我。
没问题就开始写正文。
```

**用户编辑后再次确认时，必须重新读取大纲文件**——别凭记忆。用户的编辑就是 ground truth。

### Phase 3：大纲 → 任务列表

用 TaskCreate 把大纲转成任务，**一个 H2 一章 = 一个 task**（不是一个 H3 一个 task——太碎、连贯性差）。每个任务描述格式：

```
写第 N 章「章节名」（含 M 个小节）
读: src1, src2, ..., srcK
追加到: kafka/Kafka_Learning_Notes.md
```

任务总数 ≈ H2 数量（典型 5-10 个）。

### Phase 4：按任务执行

按顺序逐 task 处理。每个 task 内部：

1. 用 Read 读该章涉及的所有源文件（一次性全读，不要碎着读）
2. **首章特别处理**：用 Write 创建主笔记 md，**第一行必须是 `# <主题> 学习笔记`**（详见下文"对接 learning-roadmap 的结构契约"）
3. 写章节标题 `## 第 N 章 · xxx`
4. **章节引言（2-3 句过渡）写进首个 `###` 小节的开头**——不要写在 `##` 和 `###` 之间，roadmap parser 不收集那段，会被悄悄丢弃
5. 按大纲里的 `###` 顺序写每个小节
6. 用 Edit / Write 追加到主笔记 md
7. TaskUpdate 标记完成 → 进入下一个 task

**风格不变量（默认全开，用户喊停才关）**：

- **详尽中文**：每个概念给出定义 → 用途 → 用例 / 注意事项，宁可啰嗦不可省略关键背景
- **适量 mermaid**：流程、架构、状态机用 `graph TD` / `sequenceDiagram` / `stateDiagram-v2`。**mermaid 节点 / 边的文字里禁用中文标点（，。；：）和英文括号**，否则解析失败——用空格或英文逗号替代
- **代码示例**：配置项、CLI、API 都给出可复制的代码块，标注语言（` ```bash`、` ```java`）
- **图文结合**：每个大块概念之后跟一个图或一段代码或一张表，不要纯文字铺 1000 字
- **小节标题用陈述句或问句**，不要照搬源文件标题（源文件常带编号 / 营销味）

### Phase 5：完成 + 联动提示

```
✓ 笔记已生成：kafka/Kafka_Learning_Notes.md
  X 章 / Y 节 / Z 个 mermaid 图 / 约 N 行

下一步建议：
  python3 ~/.skills/learning-roadmap/scripts/build_roadmap.py kafka/Kafka_Learning_Notes.md
转成可交互的 HTML 学习路线图。
```

## 对接 learning-roadmap 的结构契约

下游 [[learning-roadmap]] 的 parser 对 md 结构有 3 条**硬契约**，违反任何一条就会丢内容或污染显示。修改本 skill 时一定要保留：

1. **第一行必须是 `# <主标题>`**
   parser 用它当 roadmap 顶部标题，**也是 localStorage 进度存储 key（`roadmap:<主标题>`）**——少了它会用文件名 stem 当 key，文件改名会丢进度
2. **`##` 和首个 `###` 之间的文字会被丢弃**
   parser 在遇到 `##` 时把 `current_section` 置空，之后的文本无处可去。所以：章节引言要写进首个 `###` 的开头，别孤悬在 `##` 下方
3. **`<!-- src: -->` 必须独占一行**
   parser 的标题正则是行级贪婪匹配，行尾的 HTML 注释会被吞进 `### ` 标题里、显示成 `1.1 xxx <!-- src: ... -->`

其余结构（`###`、`####` 子标题、` ```mermaid `、` ```bash ` 等代码块、列表）parser 都能正确处理，写正文时按 markdown 常规习惯就行。

## 关键设计决策（修改 skill 时不要破坏的不变量）

1. **强制大纲 checkpoint**——这是 skill 比一次性 prompt 的最大增量。跑错 1 页大纲重写 1 分钟，跑错 5000 行正文重写半小时
2. **源文件映射用 HTML 注释**——人眼可读、Claude 二次执行可解析、不污染最终笔记的可读性（注释在渲染时隐藏）
3. **一个 H2 一个 task**——章节内部连贯性靠"一次性读完该章所有源文件 + 一次性生成"保证，不是靠后处理
4. **绝不让脚本做语义判断**——`scan_sources.py` 只做机械扫描和清洗，章节命名 / 合并 / 重组全部交给 LLM
5. **不替用户处理"材料从哪来"**——这是无限维度的问题，明确划在 skill 外
6. **默认风格全开（详尽 + mermaid + code）**——按用户偏好（详见 [Kafka 笔记 ground truth](../../git/group-question/kafka/Kafka_Learning_Notes.md)：5550 行 / 59 H2 / 179 H3 / 114 mermaid）

## 资源 / 依赖

- `scripts/scan_sources.py`：扫描脚本，纯 Python 3.10+ 标准库，无外部依赖
- 下游联动：[[learning-roadmap]] 把产出的 md 转可交互 HTML（米黄手账风 + 三态进度 + mermaid 渲染）
- 上游：用户自己负责材料收集（爬虫 / 下载 / 整理）
